# Тестовое окружение Telegram (тестовые DC)

Прокси умеет направлять трафик к **тестовым дата-центрам** Telegram
(тестовое окружение) вместо продовских. Нужно разработчикам, которые тестируют
ботов/клиентов в тестовой среде Telegram, находясь за DPI-блокировкой.

## Зачем

Бот, созданный в тестовом окружении (через BotFather на тестовом DC, либо Bot API
`.../bot<token>/test/...`), доступен только аккаунтам из того же тестового окружения.
Чтобы подключиться к нему клиентом с ПК из сети, где Telegram блокируется,
удобно гонять трафик через локальный MTProto-прокси.

Обычный маршрут прокси сюда не годится: основной транспорт — WebSocket к
`kws{dc}.web.telegram.org`, а у **тестовых DC вебсокет-фронтенда нет** (там голый
MTProto по TCP).

## Telegram Desktop: работает автоматически

Telegram Desktop помечает тестовые DC сдвигом номера +10000 (10001–10003),
поэтому прокси распознаёт их сам — включать ничего не нужно. Можно быть
авторизованным в основном и тестовом аккаунте одновременно: продовый трафик
идёт обычным маршрутом, тестовый — сразу на тестовые DC.

Тестовый трафик прокси ведёт так:

- использует адреса тестовых DC (`1: 149.154.175.10`, `2: 149.154.167.40`,
  `3: 149.154.175.117`);
- пропускает первичный WS к `web.telegram.org` и идёт сразу в fallback;
- **не** использует боевые Cloudflare-релеи (`kws{dc}.<домен>`), т.к. они знают
  только продовые DC.

## Принудительный тест-режим (`force_test_dc`)

Клиенты без сдвига +10000 (например, Telethon/TDLib через MTProto-прокси)
сообщают тестовые DC как обычные 1–3 — прокси не может отличить их от
продовых. Для них есть принудительный режим: **весь** трафик уходит на
тестовые DC, продовые аккаунты через этот прокси работать перестанут.

**GUI:** Настройки → «Датацентры Telegram (DC → IP)» → чекбокс
**«Принудительно тестовое окружение (тестовые DC)»**.

**CLI:**

```bash
tg-ws-proxy --force-test-dc      # + ваши --secret / --port
```

## Если тестовые DC заблокированы провайдером

Основной путь — прямой TCP к тестовому DC на порт 443. Если провайдер режет
и его, поднимите **свой Cloudflare Worker** и укажите его домен
(GUI: поле «Cloudflare Worker домены» / CLI: `--cfproxy-worker-domain`):
worker коннектится к тестовому DC уже с эджа Cloudflare, минуя фильтрацию.
Worker из инструкции [Cloudflare Worker](./CfWorker.md) подходит без изменений —
он релеит на произвольный `dst` из query-параметра.

## Ограничения

- В тестовом окружении существуют только DC 1-3; запрос к DC вне таблицы
  завершится в логе как `no fallback available`.
- Принудительный режим глобальный: с включённой галочкой продовые аккаунты
  через этот прокси не работают. Для параллельной работы клиента без сдвига
  +10000 запустите второй экземпляр на другом порту с `--force-test-dc`.
